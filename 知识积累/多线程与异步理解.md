# 多线程与异步——从根上理解

---

## 一、进程和线程

**进程**：操作系统分配资源的单位。每个 .exe 启动就是一个进程，有独立的内存空间。

**线程**：CPU 执行的最小单位。一个进程可以有多个线程，**共享同一块内存**。

```
进程（你的程序）
├── 线程1（主线程/UI线程）
├── 线程2（线程池线程）
└── 线程3（线程池线程）
```

多线程的核心价值：**让多件事情真正同时发生**（多核CPU下是物理并行）。

多线程的核心风险：多个线程同时读写同一块内存 → **数据竞争**，结果不可预测。

---

## 二、同步阻塞是什么问题

```csharp
// 假设这行要等3秒
var result = 数据库查询();  // 线程在这里死等
// 3秒后才能执行下一行
更新界面();
```

对于 UI 应用，主线程就是 UI 线程。它死等的这3秒，用户点按钮没反应，界面不刷新，看起来"卡死"了。

**根本原因**：一个线程同一时刻只能做一件事。

---

## 三、解决方案1：多线程（真并发）

```csharp
// 开一个新线程去做耗时操作
Thread t = new Thread(() => {
    var result = 数据库查询();  // 在新线程里等，主线程不受影响
});
t.Start();
// 主线程立刻继续，不等上面
```

主线程和新线程**真正同时运行**。

问题：线程是重量级资源，创建/销毁有开销；大量线程会让CPU在线程切换上浪费时间。

**线程池**解决了这个问题：预先创建一批线程，反复复用，不用每次新建。

```csharp
ThreadPool.QueueUserWorkItem(_ => {
    var result = 数据库查询();
});
```

## 四、解决方案2：异步（async/await）

异步的核心思想完全不同——**不是"开新线程去等"，而是"等的时候先去做别的事"**。

现实类比：

- **同步**：你去餐厅点餐，站在收银台前死等厨房做好，期间什么都不干。
- **多线程**：你雇了个助手去等，你自己去做别的事。
- **异步**：你点完餐，服务员给你个号码牌，你去座位上玩手机。餐好了服务员叫你。

```csharp
async Task 点餐流程()
{
    下单();                    // 正常执行
    var 餐 = await 等待出餐(); // 挂起，把线程还给别人用
                               // 出餐后，从这里恢复继续
    吃饭(餐);
}
```

`await` 的本质：**把后续代码注册成回调，当前线程立刻去做别的事**。没有线程在傻等。

## 五、async/await 和多线程的关系

这是最常见的误解：**`async/await` 不等于多线程**。

| | 是否创建新线程 | 适合场景 |
|---|---|---|
| `async/await` | 不一定 | **IO密集**：网络、数据库、文件读写（等待期间没有CPU计算） |
| `Thread` / `Task.Run` | 是 | **CPU密集**：图像处理、加密运算、大量计算 |

网络请求为什么适合异步而不是多线程？

```
发出网络请求 → 等待网络响应（这段时间CPU什么都不用做）→ 收到数据处理
```

等待网络的时间里，CPU是空闲的。用多线程去"等"纯粹是浪费线程资源。异步让这段时间线程去做别的请求，一个线程可以同时"在等"成千上万个网络请求。

## 六、状态机——async/await 的真相

编译器把 `async` 方法变成一个**状态机**：

```csharp
async Task 示例()
{
    Console.WriteLine("第1步");
    await Task.Delay(1000);     // ← 状态0结束
    Console.WriteLine("第2步"); // ← 状态1开始
    await Task.Delay(1000);     // ← 状态1结束
    Console.WriteLine("第3步"); // ← 状态2开始
}
```

编译后等价于：

```
状态0：打印"第1步"，启动计时器，注册"1秒后进入状态1"，返回
状态1：打印"第2步"，启动计时器，注册"1秒后进入状态2"，返回
状态2：打印"第3步"，完成
```

没有线程在等待，只有一个计时器在后台，时间到了通知状态机继续。

## 七、线程安全——共享内存的危险

多线程共享内存时，必须保护共享数据：

```csharp
// 两个线程同时执行 count++
// count++ 实际是三步：读 → 加1 → 写
// 线程A读到0，线程B读到0，两个都写1，最终 count=1 而不是2
int count = 0;
```

**lock**：同一时刻只允许一个线程进入

```csharp
private readonly object _lock = new();
int count = 0;

void 增加()
{
    lock (_lock)  // 其他线程在此等待
    {
        count++;  // 安全
    }
}
```

**原则**：读写共享状态的代码必须在 `lock` 里。

## 八、Task 是什么

`Task` 是"一件将来会完成的事"的句柄，可以：

```csharp
Task t = Task.Run(() => 耗时操作());

t.Wait();               // 同步等待它完成（会阻塞当前线程）
await t;                // 异步等待它完成（不阻塞当前线程）
bool done = t.IsCompleted;  // 查询是否完成
```

`Task<T>` 带返回值：

```csharp
Task<int> t = Task.Run(() => 计算结果());
int result = await t;  // 拿到计算结果
```

## 九、CancellationToken——协作式取消

取消一个正在运行的异步操作，靠的是双方协作，不是强制中断：

```csharp
// 创建者：持有取消控制权
var cts = new CancellationTokenSource();

// 启动任务，把 token 传进去
Task.Run(() => 长跑循环(cts.Token));

// 需要取消时
cts.Cancel();  // 发出信号
```

```csharp
// 长跑循环内部：主动检查信号
void 长跑循环(CancellationToken ct)
{
    while (!ct.IsCancellationRequested)  // 每轮检查
    {
        做一件事();
    }
    // 或者
    await Task.Delay(1000, ct);  // Delay 也感知取消，直接抛异常
}
```

类比：取消按钮不是"强制关门"，而是"举牌子"，循环里的代码看到牌子主动停下来。

## 十、一张图总结

```
耗时操作
    │
    ├── IO密集（网络/磁盘）→ async/await → 线程挂起，不占CPU，完成后恢复
    │
    └── CPU密集（计算）→ Task.Run → 推到线程池，真正并行执行

共享数据 → lock 保护 → 同一时刻只有一个线程能改

需要停止 → CancellationToken → 协作式，循环主动检查后退出

结果要回UI线程 → Dispatcher.InvokeAsync → 投递到UI消息队列
```
